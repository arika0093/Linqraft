# 01. System Overview

## 1. Product definition

Linqraft is a compile-time LINQ projection toolchain centered on the `SelectExpr` authoring model.
Its documented purpose is to let developers write projection code in ordinary LINQ syntax while gaining:

- DTO generation driven by the query body
- support for null-conditional authoring in expression-tree scenarios
- zero runtime mapping dependencies
- Roslyn analyzers and code fixes that help migrate and maintain projection code

In practical terms, Linqraft behaves as a Roslyn source generator plus analyzer suite whose output is ordinary C# projection code.

## 2. Core user-visible capabilities

### 2.1 Query-based DTO generation

The projection body is the source of truth.
For the explicit DTO workflow, a user writes an anonymous-shape selector such as:

```csharp
query.SelectExpr<Order, OrderDto>(o => new
{
    o.Id,
    CustomerName = o.Customer?.Name,
    Items = o.Items.Select(i => new
    {
        i.ProductName,
        i.Quantity,
    }),
});
```

The clone MUST generate:

- a root DTO type named by the caller (`OrderDto`)
- nested DTO types for nested anonymous objects and nested anonymous collection items
- member types matching the authored projection shape

### 2.2 Null-conditional authoring for expression trees

The library docs treat `?.` authoring as a first-class feature.
For `IQueryable`, a clone MUST accept authored null-conditional expressions and emit a provider-compatible projection that is semantically equivalent to explicit null checking.

### 2.3 No runtime mapping framework

Generated code is standard LINQ projection code.
The clone MUST NOT require consumers to register profiles, use runtime reflection-based mapping, or ship a runtime mapper assembly to production in order to execute the projected query.

### 2.4 IDE tooling and migration support

Linqraft is not only a generator.
It also includes analyzers and code fixes that:

- suggest replacing `Select` with `SelectExpr`
- convert anonymous shapes into DTOs
- enforce explicit capture of outer variables
- warn against depending on unstable generated nested DTO names

## 3. Supported execution models

The documented surface spans three execution models:

1. **`IQueryable<T>` projections**  
   This is the flagship scenario, especially for EF Core and other expression-tree providers.

2. **`IEnumerable<T>` projections**  
   The same authoring style is supported for in-memory collections.

3. **Generated mapping methods**  
   An optional beta workflow emits reusable projection methods for compiled queries, precompiled queries, and code organization scenarios where interceptor-driven call-site replacement is not preferred.

## 4. Component model

## 4.1 Consumer-authored source

The consumer writes one of the documented `SelectExpr` patterns or one of the mapping-method declaration patterns.
The authored selector body is the canonical specification of the projection.

## 4.2 Embedded support surface

The generator emits a small support surface into the consumer compilation, including:

- `SelectExpr` dummy extension methods
- `LinqraftMappingGenerateAttribute`
- `LinqraftMappingDeclare<T>`
- `LinqraftAutoGeneratedDtoAttribute`

These support types exist so that consumer code compiles before generation is completed.
The dummy `SelectExpr` methods MUST NOT be the runtime implementation path for a successful build.

## 4.3 Source generator

The generator is responsible for:

- reading global MSBuild options
- discovering `SelectExpr` invocations
- discovering mapping declarations
- generating DTO types
- generating interceptor-backed replacement methods for ordinary `SelectExpr` usage
- generating reusable mapping methods for the mapping-method workflows

## 4.4 Analyzer and code-fix layer

The analyzer layer serves two roles:

1. **adoption tooling**  
   It helps users migrate from plain `Select` and anonymous shapes to Linqraft patterns.

2. **correctness guardrails**  
   It flags constructs that are unsafe or unsupported for generation, such as missing capture variables or direct use of unstable hash-namespaced DTO types.

## 4.5 Generated artifacts

The generated artifact set can include:

- interceptor methods bound to specific `SelectExpr` call sites
- DTO types
- nested DTO types
- mapping extension methods
- embedded support types and attributes

When `EmitCompilerGeneratedFiles` is enabled, the docs show these artifacts appearing under a `Generated/Linqraft.SourceGenerator/` folder.

## 5. End-to-end lifecycle

The documented lifecycle for a normal `SelectExpr` flow is:

1. the developer writes a projection using `SelectExpr`
2. analyzers surface migration suggestions or correctness issues in the IDE
3. during compilation, the generator reads the code and current global properties
4. the generator emits DTO types and projection implementation code
5. the authored `SelectExpr` call is redirected to generated code
6. the resulting runtime execution is a normal LINQ `Select` projection over `IQueryable` or `IEnumerable`

For the mapping-method flow, step 4 emits a reusable extension method instead of relying on a direct call-site interceptor.

## 6. Architectural invariants

A compatible clone MUST preserve the following system-level invariants:

### 6.1 Source-first mapping

The query body, not an external configuration registry, defines the produced DTO shape.

### 6.2 Generated output is plain C#

Generated code MUST be readable ordinary C# that can be inspected through IDE navigation or emitted generated files.

### 6.3 Stable root DTO contract, unstable auto-generated nested implementation detail

Root DTOs created through the explicit DTO workflow are part of the user-facing contract.
Automatically generated nested DTO names are intentionally implementation details and MUST be treated as unstable across shape changes.

### 6.4 Ordinary LINQ provider execution

The clone MUST translate projections into plain LINQ constructs that execute through the underlying provider rather than through a custom runtime evaluator.

### 6.5 Partial-type extensibility

Generated DTOs MUST remain partial so that downstream code can extend them without forking the generator.

## 7. Non-goals

The current documentation does not position Linqraft as:

- a general-purpose runtime object mapper
- an attribute-driven mapper where the attribute model is the primary source of truth
- a replacement for arbitrary query composition outside projection authoring
- a tool for non-LINQ data shaping

A compatible clone SHOULD keep the product focused on projection authoring and generation rather than broadening into a different class of mapper.
