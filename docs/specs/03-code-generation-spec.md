# 03. Code Generation Specification

## 1. General generation responsibilities

A compatible clone MUST generate ordinary C# source that fulfills four responsibilities:

1. provide compile-time support types so consumer code compiles
2. translate documented `SelectExpr` authoring forms into executable LINQ projections
3. generate DTO types when required
4. emit reusable mapping methods for the optional mapping-generation workflows

## 2. Embedded support types

The generator MUST emit a support surface equivalent in role to the current one:

- dummy `SelectExpr` extension methods
- `LinqraftMappingGenerateAttribute`
- `LinqraftMappingDeclare<T>`
- `LinqraftAutoGeneratedDtoAttribute`

### 2.1 Dummy `SelectExpr` behavior

The emitted stub methods exist only so the user's source compiles before generation.
If one of these dummy methods is executed directly, it MUST throw with a clear "source generator should have replaced this" style message.

This is a compatibility requirement because the current docs and source treat direct execution as a bug condition.

## 3. Interceptor-backed projection generation

For ordinary `SelectExpr` usage, the clone MUST generate code that effectively turns the authored call into a typed LINQ `Select` projection.

## 3.1 Generated method shape

The current documented/generated model has these properties:

- generated methods live in the `Linqraft` namespace
- generated methods are placed in a file-scoped or file-local helper such as `GeneratedExpression`
- each generated method is unique per call site
- `InterceptsLocationAttribute` is used to bind a generated method to a specific source location

The exact helper type name MAY differ internally, but a compatible clone SHOULD preserve the same observable behavior:

- each `SelectExpr` call site resolves to generated code
- the generated method is uniquely associated with the call site
- the generated method calls ordinary LINQ `Select`

## 3.2 Generated method responsibilities

For each intercepted projection, the generated method MUST:

1. identify the concrete source type
2. cast the incoming `query` to the real `IQueryable<TSource>` or `IEnumerable<TSource>`
3. project into the correct target shape using standard LINQ `Select`
4. return the result in the originally expected generic result form

The current generated examples use double `as object` casts around the source and result.
A clone MAY change that exact implementation detail as long as it preserves the same compiled behavior.

## 4. DTO generation

## 4.1 Root DTO generation

When the explicit DTO pattern is used, the clone MUST generate a root DTO declaration whose:

- name matches the caller-supplied type argument
- namespace matches the intended destination namespace
- members match the authored selector shape
- declaration is `partial`

The current docs/examples model the root DTO as a partial class by default.

## 4.2 Nested DTO generation

When the selector contains nested anonymous objects or anonymous collection item projections, the clone MUST generate additional DTO types.

Documented expectations:

- nested DTOs are derived from the parent property name
- a hash component is added to avoid collisions
- the default naming strategy uses a hash-based namespace segment such as `LinqraftGenerated_HASH`
- an alternate strategy uses hash-suffixed class names instead of a separate namespace

## 4.3 Marker attributes on generated DTOs

Automatically generated nested DTOs MUST be marked so analyzers can recognize them as implementation details.
The current contract uses:

- `EditorBrowsable(EditorBrowsableState.Never)`
- `LinqraftAutoGeneratedDtoAttribute`

The root explicit DTO SHOULD NOT be marked as an auto-generated nested implementation-detail type.

## 4.4 Class vs record generation

The root and nested DTO declarations MUST honor the global configuration that selects class or record generation.

## 4.5 Property generation

For each projected member, the clone MUST generate a corresponding property with:

- the projected member name
- the projected member type
- the configured accessor pattern
- the configured use or non-use of the `required` keyword

### Member-name rules

The following behaviors are required:

- `o.Id` becomes `Id`
- `CustomerName = o.Customer?.Name` becomes `CustomerName`
- nested collection properties use the parent member name (`Items`, `ChildDtos`, and so on)

## 4.6 Property suppression by predefinition

If the user predefines a property with the same name in a partial declaration of the DTO, the clone MUST NOT emit a duplicate property for that member.
This is required to preserve the documented "control property accessibility yourself" workflow.

## 5. Null-conditional translation rules

For `IQueryable`, the docs require that `?.` authoring be converted into provider-compatible explicit null checks.

### 5.1 Simple member access

Input:

```csharp
o.Customer?.Name
```

Required output semantics:

```csharp
o.Customer != null ? o.Customer.Name : null
```

### 5.2 Chained member access

Input:

```csharp
o.Customer?.Address?.Country?.Name
```

Required output semantics:

```csharp
o.Customer != null
    && o.Customer.Address != null
    && o.Customer.Address.Country != null
        ? o.Customer.Address.Country.Name
        : null
```

### 5.3 Recursive application

The translation MUST apply recursively inside:

- root DTO property initializers
- nested anonymous objects
- nested named DTO object initializers
- collection item selectors

### 5.4 `IEnumerable<T>` behavior

For `IEnumerable<T>`, the same authored syntax MUST remain valid.
Because in-memory delegates can already execute `?.`, a clone MAY avoid the explicit rewrite as long as the user-visible behavior is preserved.

## 6. Collection and nested projection rules

The docs treat nested collections as a first-class feature.
The clone MUST support collection expressions such as:

```csharp
Items = o.OrderItems.Select(oi => new
{
    ProductName = oi.Product?.Name,
    oi.Quantity,
})
```

Required behavior:

- the parent DTO property type MUST be a collection type compatible with the authored projection
- nested anonymous collection elements MUST become generated nested DTO types
- nested named DTO elements MUST retain the named DTO type
- generated mapping code MUST recurse into the nested projection rather than treating the collection as opaque

The documented collection story also covers flattening-oriented operators in related rules such as array nullability removal.
A compatible clone SHOULD therefore support both `Select` and `SelectMany` as authorable collection-shaping operators when they appear in otherwise valid projection expressions.

## 6.1 Calculated-field support

The README and usage examples treat calculated members as a normal part of the projection model.
A compatible clone MUST allow ordinary LINQ-compatible calculated-field expressions to pass through unchanged except for any required null-conditional rewriting.

Documented examples include aggregate operators such as:

- `Sum`
- `Max`
- `Min`
- `Count`
- `Average`

## 7. Array nullability removal

The library docs define an explicit convenience rule for collection-shaped properties.
When all of the following are true:

1. the expression does not use an explicit ternary operator
2. the result type is `IEnumerable<T>` or a derived collection type
3. the expression uses null-conditional access
4. the expression contains `Select` or `SelectMany`

the clone MUST:

- remove the nullable marker from the generated collection property type
- emit a default empty collection value rather than `null` in the "source is null" branch

The empty value SHOULD match the projected collection shape:

- `Enumerable.Empty<T>()` is appropriate for interface-shaped enumerable outputs
- `new List<T>()` is appropriate for `List<T>` outputs
- another equivalent empty value is acceptable if it matches the generated property type and preserves the documented "prefer empty over null" contract

When any of those conditions are not met, the clone MUST preserve the natural nullability of the authored expression.

## 8. Capture handling in generated code

When the selector uses the `capture:` parameter, the generated code MUST:

- accept an extra argument representing the capture object
- recover the captured values by name
- make them available to the projection logic as ordinary locals or equivalent values

The current docs show this as object-based capture rehydration in generated code.
A clone MAY choose a different internal representation as long as:

- authoring stays `capture: new { ... }`
- generated semantics are equivalent
- analyzer behavior remains compatible

## 9. XML documentation propagation

When comment generation is enabled, the clone MUST propagate supported documentation from source members onto generated DTO declarations and properties.
Supported sources are documented as:

- XML summary comments
- EF Core `[Comment("...")]`
- `[Display(Name = "...")]`
- single-line comments

The clone MUST treat all four source kinds as first-class supported inputs rather than as best-effort extras.

The output modes are specified in the configuration spec.

## 10. Nested `SelectExpr` generation

For nested explicit DTO generation:

- the clone MUST use the explicitly named nested DTOs rather than creating hash-generated anonymous nested DTOs
- the clone MUST rely on empty partial declarations to determine that those DTO types exist and where they belong
- the clone MUST fail analysis or generation if the required declarations are missing, because the docs state this scenario cannot be resolved correctly otherwise

## 11. Mapping-method generation

The optional mapping-method workflows MUST generate reusable extension methods instead of interceptor-bound call-site replacements.
This feature exists specifically to support scenarios such as EF Core compiled queries, EF Core precompiled queries, and NativeAOT-sensitive code paths where reusable generated methods are preferable to direct interceptor use.

### 11.1 Helper-class pattern

Expected behavior:

- a generated static partial class is created
- the class name includes a hash suffix to avoid collisions
- a projection method is generated with the default name `ProjectTo{EntityName}` or the custom name supplied by the attribute
- the method executes the same projection semantics as the corresponding `SelectExpr` query

### 11.2 Static partial class pattern

Expected behavior:

- the method is generated into the user-declared static partial class
- the declared template method serves only as a specification source and SHOULD NOT be the runtime entry point

## 12. Generated-file observability

A compatible clone SHOULD continue to produce human-inspectable generated C# with:

- an auto-generated file header
- nullable context enabled
- warning handling suitable for generated code

When consumers enable `EmitCompilerGeneratedFiles`, they SHOULD be able to inspect per-projection output in a generated folder similar to the one shown in the current FAQ.
The documented emitted-file convention is `Generated/Linqraft.SourceGenerator/SelectExpr_HASH.g.cs`, and a compatible clone SHOULD preserve that general folder layout unless the public docs are intentionally updated.
